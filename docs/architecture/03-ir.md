# Intermediate representation

**Status**: v0.2 (draft, 2026-05-09). v0.2 drops the separate `NodeKind` discriminator type; op identity is carried by a single `Op` enum with primitive variants and one `Op::Fused(id, params)` arm. v0.1 used a `NodeKind::Primitive | Fused` wrapper.

The data structures that carry a model from definition through optimization to execution. This is the *what* of fuel's IR: the shape of the DAG, the op vocabulary, how fused ops extend it without enlarging it, where layout lives, and how the canonical primitive form coexists with the optimized form.

The IR is the surface every other component reads from and writes to. Backends consume it, the optimizer transforms it, the runtime executes it. When this section's claims change, every other section may need revision.

---

## The DAG

A model is a directed acyclic graph of operations. The graph is an arena: nodes are added by index, never removed, never re-indexed. Edges are encoded as input-id lists on each node — `node.inputs: Vec<NodeId>`. A `NodeId` is a `usize`-newtype into the arena.

Three properties are non-negotiable:

- **Immutable nodes after creation.** A node's op, inputs, shape, and dtype are fixed at construction. If an optimizer wants to change a node, it inserts a new node and rewrites the edges of consumers to point to the new id. This makes graph rewrites tractable and lock-free reading possible.
- **Append-only arena.** Optimization grows the arena; orphaned nodes are not removed. Garbage-collection of unreachable nodes can happen in batch (e.g., between optimization passes) but is not architectural — it's a memory hygiene concern, not a correctness one.
- **Side-table metadata.** Information that's derivable from the graph (layouts of view ops) or that varies independently of node identity (tolerance budgets, route-specific kernel assignments, profile data) lives in side tables keyed by NodeId, not on the node itself. The node is the structural fact; side tables are interpretations.

The graph is wrapped in `Arc<RwLock<_>>` so handles can cross thread boundaries. Reading is uncontended in steady state (the optimizer commits new graph versions atomically; readers see whichever version they hold an `Arc` to).

## The op vocabulary

The IR has two layers of op identity, each with a different evolution model:

**The `Op` enum is closed and small.** Roughly 80-90 variants covering arithmetic primitives (Add, Mul, Exp, MatMul), shape/view ops (Transpose, Permute, BroadcastTo, Reshape, Slice, Unsqueeze), reductions (SumDim, MaxDim, ReduceSumTo, ReduceMaxTo), comparison family (Equal, Less, Greater, ...), memory/control (Const, Copy, Move, Release), and casts. Sized comparable to stablehlo's primitive set. New primitives are rare events — the enum is exhaustively matched in many places, and additions force review across all of them. This is the fixed vocabulary.

**The fused-op registry is open and extensible at build time.** Anything that's a meaningful composition of primitives lives in `FusedOpRegistry` rather than in `Op`. SoftmaxLastDim, RmsNormLastDim, LayerNormLastDim, Rope, FusedLinear, Conv2D, ConvTranspose2D, FlashAttn, PagedAttn, QMatMul, plus their fused-backward helpers — these are registry entries, not Op variants. Each entry carries:

- A canonical primitive subgraph it represents (for fusion pattern recognition).
- A decomposition function (for lowering to the base map; see below).
- Per-backend kernel implementations with cost annotations (for placement and route selection).
- A backward identity (another registry entry, or "decompose first then differentiate").
- Shape and dtype rules.
- Tolerance/error annotations per backend implementation.

Adding a fused op is one registry entry plus a kernel function; no `Op` enum edit, no autograd edit, no executor arm edit. The registry is populated at startup and frozen thereafter — runtime extensibility is not a goal.

The two layers exist because they evolve at different rates. Primitives are infrastructure; fused ops are accumulating optimization wins. Conflating them (the historical `Op` hybrid) makes every fused-op addition a multi-file edit. Separating them makes fused-op extension cheap.

## How nodes carry their op identity

A single `Op` enum carries all op identity. Most variants are primitives; one arm — `Op::Fused(id, params)` — delegates to the open registry. There is no separate `NodeKind` discriminator type.

```rust
pub enum Op {
    // ~80 primitive variants, closed and exhaustive.
    Add,
    Sub,
    Mul,
    Div,
    MatMul,
    Conv2D { stride: (usize, usize), padding: (usize, usize), groups: usize },
    BroadcastTo(Shape),
    Permute(Vec<usize>),
    Slice { dim: usize, start: usize, end: usize, step: usize },
    Cast(DType),
    // ... etc.

    // One arm for fused ops; their parameters live in a registry-side type.
    Fused(FusedOpId, FusedOpParams),
}

pub struct Node {
    pub op:     Op,
    pub inputs: Vec<NodeId>,
    pub shape:  Shape,
    pub dtype:  DType,
}
```

The two-tier shape inside one type:

- **Primitive variants** (`Op::Add`, `Op::Conv2D { stride, padding, groups }`, ...) — pattern-match with variant payloads. Per-instance structural data (Conv2D's stride, BroadcastTo's target shape, Slice's geometry) lives on the variant. Consumers handle these with specific match arms or `_` catch-alls.
- **`Op::Fused(id, params)`** — the registry-extended arm. The id identifies which fused op (SoftmaxLastDim, FlashAttn, etc.); the params carry per-instance fused-op data (FlashAttn's softmax_scale, etc.). Consumers either delegate uniformly to the registry (autograd, executor's pre-resolved KernelRef path) or pattern-match on specific fused ids when needed (the few rules that anchor on a single fused-op).

Both forms are first-class in the IR. A fused-op node and the equivalent primitive subgraph are interchangeable representations of the same computation; the optimizer moves freely between them.

The split exists because primitives and fused ops evolve at different rates. Primitives are infrastructure (closed enum, rare additions, every consumer reviewed when one is added). Fused ops are accumulating optimization wins (open registry, frequent additions, registered without touching the `Op` enum beyond the existing `Fused` arm).

## Layout: a side-table, not metadata on storage

Tensor layout (strides, broadcast factors, slice offsets, transpose permutations) is graph-side, not storage-side. A `Layout` lives in `Graph.layouts: HashMap<NodeId, Layout>`, populated automatically when a view op is added.

This decoupling means view ops are zero-copy: `Op::Transpose`, `Op::Permute`, `Op::BroadcastTo`, `Op::Reshape`, `Op::Slice`, `Op::Unsqueeze` produce nodes whose storage *aliases* their input's bytes; the view-output's `Layout` describes how to interpret those bytes for downstream consumers. No new bytes are allocated; no copy is made.

When a kernel needs a layout it can't handle (e.g., a kernel that requires contiguous input is fed a strided view), the optimizer inserts an explicit `Op::Contiguize` node (or its equivalent) — a *materialization* op that allocates new bytes in the requested layout. The optimizer is the place where "this kernel can't handle stride X, so insert a fixup" is decided; the executor never silently materializes.

Kernels advertise via `KernelCaps` what layouts they accept (`strided_input: bool` and future capability flags). The optimizer reads these and inserts materialization ops only when needed, never speculatively.

## Tolerance metadata

Tolerance budgets attach to the IR as side-table metadata, keyed by NodeId or by NodeId range (for subgraph-level budgets). See [07-tolerance](07-tolerance.md) for the budget model; here, two structural notes:

- **Budgets are propagated, not embedded.** A user setting a graph-level default doesn't tag every node; the optimizer reads the hierarchical specification (graph default → subgraph override → per-op override) when it needs to know the budget for a candidate rewrite.
- **Per-call overrides don't mutate the IR.** A `realize(tolerance: ...)` override is communicated to the route picker, not written back into the graph. The graph's tolerance is the model author's intent; per-call overrides are the caller's adaptation.

## The base map: fully-decomposed primitive DAG, permanently retained

When a model is loaded or built, the resulting DAG is whatever the user wrote — a mix of primitive ops, fused-op nodes for compositions the user invoked directly, and possibly some half-fused intermediate forms. This is the *user-facing form*.

Decomposition produces the **base map**: every `Op::Fused(id, params)` node is replaced by the registry entry's primitive-subgraph decomposition, recursively, until no fused-op nodes remain. The base map contains only primitive `Op` variants — no `Op::Fused` arms. It is the canonical, fully-elaborated form of the model — every operation expressed in terms of the closed primitive vocabulary.

The base map is **retained as a permanent artifact** alongside the optimized form. This is a deliberate choice with concrete consequences:

- **Re-optimization from canonical state.** When backend telemetry changes (new device available, memory pressure shifted, profile data refined), re-optimizing from the base map is straightforward — it's the canonical input. Re-optimizing from the previous optimized form would require undoing fusions and reorderings before redoing them.
- **Pattern harvesting against canonical form.** The base map is what pattern-harvest telemetry (see [08-pattern-harvest](08-pattern-harvest.md)) reads to identify long unfused sequences. Without the base map, harvesting would see whatever optimizations had already fired, biasing the data toward what fuel already supports.
- **Bisection during debugging.** A user comparing strict vs. fuzzy outputs can compare against the base map's strict execution to isolate which optimization introduced unexpected divergence.
- **Decomposition is one-way; the base map is the well-defined input to optimization.** The optimizer doesn't have to invert decompositions; it builds the optimized form by transforming the base map, leaving the base map untouched.

The cost is roughly 2× graph storage. For typical inference graphs (low thousands of nodes), this is negligible. For pathological cases (millions of nodes), the policy can be revisited; the architectural commitment is that the base map *is* a retained artifact, not that it must be retained in fully-materialized form forever.

## The optimized form: top-N routes with pre-resolved kernels

The base map feeds the optimizer. The optimizer produces the **optimized form**, which is one or more candidate routes (default N=3, configurable) ranked by cost.

Each route is a graph derived from the base map via lowering rewrites, fusion rewrites, algebraic equivalences, dtype lowering (under tolerance), placement decisions, transfer-op insertions, and layout fixups. Routes share whatever subgraphs they have in common; routes diverge where the optimizer found genuinely different alternatives. The set of routes is preserved through to runtime, where the route picker chooses one per execution based on current telemetry.

Within each route, **every node carries a pre-resolved `KernelRef`** for the backend it's been placed on, plus enough metadata for execution:

- `KernelRef` — the function pointer to call.
- Backend and device assignment — which backend instance owns this op's execution.
- Cumulative error bound along the route — what's been spent against the tolerance budget.
- Cost estimate — what the optimizer thinks this op costs (used by the route picker for tiebreaking).

The executor reads the pre-resolved data and dispatches; it does not look anything up. The binding table — the map from `(op, dtypes, backend)` to `KernelRef` — is consulted by the optimizer at planning time to populate the per-node `KernelRef`s. At execution time, the binding table is irrelevant.

This is the answer to one of the architectural-audit cross-cutting questions: the binding table is a planning-time catalog, not a runtime lookup. Today's code consults the binding table at execution; that's a current shape, not a fundamental requirement, and it changes during the architecture's normalization to this section.

The benefits of pre-resolution:

- **Hot-path simplicity.** Executor's per-op work is "call function pointer with these inputs and outputs" — no hashmap lookup, no dispatch-table indirection.
- **Placement is honored.** A route's kernel assignment is bit-stable from optimizer commit through execution. There's no execution-time logic that could second-guess what the optimizer decided.
- **Multiple routes with different kernel choices coexist cleanly.** Route A's matmul node carries cuBLAS's KernelRef; Route B's same matmul node (different NodeId, conceptually-equivalent computation) carries an AOCL KernelRef. They share inputs, diverge in execution, neither requires runtime dispatch logic.

## How the user-facing form, base map, and optimized form relate

```text
user-facing form  ──(decomposition)──→  base map  ──(optimization)──→  optimized form
   (mixed prim+fused,                   (primitive only,                (top-N routes,
    whatever user wrote)                 canonical, retained)            pre-resolved kernels)
```

Three artifacts. All three live in the graph state as long as the model is in use.

- The **user-facing form** is what `Tensor::matmul()`, `Tensor::softmax_last_dim()`, etc. directly produce. It carries fused-op nodes when the user invoked fused ops; it's the user's mental model of their computation.
- The **base map** is the canonical reference. Decomposition is mechanical and deterministic; given a user-facing form, there is exactly one base map.
- The **optimized form** is what executes. There are many possible optimized forms for a given base map; the optimizer commits the top-N it can find within its budget.

When the user calls `realize()`, the runtime route picker reads the optimized form and chooses one of the N candidate routes based on per-call tolerance, current backend telemetry, and (in the future) request priority. The chosen route executes; the others remain available for the next `realize()` call.

When backend telemetry changes meaningfully or new profile data arrives, the optimizer re-runs against the (unchanged) base map and produces a fresh optimized form; the new form replaces the old via atomic Arc swap. Runs in flight finish on the old form; new runs start on the new form.

## What lives where: a quick reference

| Concept | Lives in | Mutable? |
|---------|----------|----------|
| Op enum (primitives) | fuel-graph | Closed; rare additions |
| Fused-op registry | fuel-graph (metadata) + fuel-storage (BackendImpl payloads) | Frozen at startup |
| User-facing graph | `Arc<RwLock<Graph>>` | Append-only |
| Base map | side-table on graph or sibling graph | Built once per model |
| Optimized form (N routes) | sibling graphs, atomic-swappable | Replaced wholesale by optimizer commits |
| Layout side-table | `Graph.layouts` | Auto-populated on view-op push |
| Tolerance metadata | side-table keyed by NodeId / NodeId range | Set at graph build, immutable thereafter |
| Per-route KernelRef + placement | side-table per optimized-form route | Pre-resolved by optimizer; immutable in route |
| Per-route cumulative error bound | side-table per optimized-form route | Pre-computed by optimizer |
| Per-route cost estimate | side-table per optimized-form route | Pre-computed; refreshed by Judge as profile data accumulates |

Implementation detail (specific Rust types, file layout, commit-and-swap mechanics) lives in phase docs. This section pins the architectural shape: three artifacts in coexistence, side-tables keyed by NodeId for everything that varies across routes or interpretations, pre-resolution at the boundary between optimizer and executor.
