It superseded v0.3's "optimized form = top-N *separate* sibling route-graphs, every node pre-resolved, replaced wholesale by atomic Arc swap." The optimized form is now a single **multi-path graph** carrying bounded Pareto-frontier paths with **decision points at branches**, kernel resolution is optional-not-mandatory, and the graph is the **input-independent model** built at load/import (not at `forward()`), serving many sessions via per-session storage. v0.3 clarified the fused-op registry cross-crate split; v0.2 dropped the `NodeKind` discriminator (op identity is one `Op` enum + an `Op::Fused(id, params)` arm).

The data structures that carry a model from definition through optimization to execution. This is the *what* of fuel's IR: the shape of the DAG, the op vocabulary, how fused ops extend it without enlarging it, how storage attaches per session, where layout lives, and how the canonical primitive form (base map) coexists with the optimized multi-path graph (the plan).

The IR is the surface every other component reads from and writes to. Backends consume it, the optimizer transforms it, the runtime executes it. When this section's claims change, every other section may need revision.

## The DAG

A model is a directed acyclic graph of operations. The graph is an arena: nodes are added by index, never removed, never re-indexed. Edges are encoded as input-id lists on each node — `node.inputs: Vec<NodeId>`. A `NodeId` is a `usize`-newtype into the arena.

Four properties are non-negotiable:

- **The graph is the model, independent of any input.** It is built when a model is loaded or imported — **not** when a forward pass runs — and is ideally fully optimized before the first input arrives. Inputs and per-session state flow *through* the persistent graph (via storage; see below); the graph itself is not rebuilt per input, per token, or per session. "No graph until `forward()` runs" is a defect this architecture corrects.
- **Immutable nodes after creation.** A node's op, inputs, dtype, and shape are fixed at construction — where "shape" is the node's *static shape expression*: a concrete dim vector, or (for data-dependent ops) a **capacity/upper-bound shape** whose true valid extent is carried at runtime by a separate count, never by mutating the node (see [Data-dependent output shapes](#data-dependent-output-shapes)). An optimizer that wants to change a node inserts a new node and rewrites consumers' edges to the new id. This makes graph rewrites tractable and lock-free reading possible.
- **Append-only arena, with compaction.** Optimization is append-only — rewrites add nodes and rewire consumers; superseded nodes are left orphaned. Because `optimize_graph`'s pathfinders explore many candidate paths the optimizers then discard, the arena accumulates substantial orphaned debris, so **compaction is required, not optional**: a compaction pass drops unreachable nodes, keeping only the base map plus the surviving optimized multi-path graph. It runs at least **before finalize-to-disk** (so the persisted `.fuel` holds the lean result, not exploration debris) and optionally **between optimization rounds** (to bound the in-memory / disk-backed working set during a long search). This is not a *correctness* concern — orphans never change results — but under the heavy-exploration optimizer it is very much a *size* concern, on disk especially (RAM less so: orphaned pages of an mmap'd graph simply never fault in).
- **Side-table metadata.** Information derivable from the graph (view-op layouts) or varying independently of node identity (tolerance budgets, storage, per-path kernel assignments, profile data) lives in side tables keyed by NodeId (and, for session state, by `(NodeId, SessionId)`), not on the node. The node is the structural fact; side tables are interpretations and instances.

The graph is wrapped in `Arc<RwLock<_>>` so handles cross thread boundaries. Reading is uncontended in steady state; many concurrent sessions read one shared, immutable-after-optimization graph.

## The op vocabulary

The IR has two layers of op identity, each with a different evolution model:

**The `Op` enum is closed and build-time-closed.** Roughly 80-90 variants covering arithmetic primitives (Add, Mul, Exp, MatMul), shape/view ops (Transpose, Permute, BroadcastTo, Reshape, Slice, Unsqueeze), reductions (SumDim, MaxDim, ReduceSumTo, ReduceMaxTo), the comparison family (Equal, Less, Greater, ...), memory/control (Const, Copy, Move, Alloc, ZeroFill, WriteSlice, Release), and casts. Sized comparable to stablehlo's primitive set. New primitives are rare events — the enum is exhaustively matched in many places. This is the fixed vocabulary. The primitive basis is a **build-time-closed Rust enum**: there is **no generic opaque / `Custom` node** in the lazy graph, so an externally-supplied (provider / JIT) op **cannot become a new primitive at runtime** — it must decompose into the existing primitive basis (carrying a `pattern` to re-fuse it) or prompt a Fuel-side **build-time** `Op`-enum extension (per the 2026-06-20 decision G3, [10-decisions-log §2026-06-20](10-decisions-log.md)).

**Each fused op carries a two-direction recipe; the registry is build-time-extensible, with a trusted runtime carve-out.** Anything that is a meaningful composition of primitives lives in `FusedOpRegistry` rather than in `Op`: SoftmaxLastDim, RmsNormLastDim, LayerNormLastDim, Rope, FusedLinear, Conv2D, ConvTranspose2D, FlashAttn, PagedAttn, QMatMul, plus fused-backward helpers. Each entry carries a **recipe in two inverse directions — both mandatory**: a `decompose` (the *break-down* — fused → primitive subgraph, for lowering to the base map) and a `pattern` (the *build-up* — the canonical primitive subgraph, for recognizing and re-fusing it). A fused op missing either half is an **opaque island** — invisible to base-map analysis and impossible to re-fuse (per the 2026-06-20 decision G1, [10-decisions-log §2026-06-20](10-decisions-log.md)). Entries also carry per-backend kernels with cost annotations, a backward identity, shape/dtype rules, and tolerance/error annotations. Adding a fused op is one registry entry plus a kernel — no `Op` enum edit, no autograd edit, no executor arm edit.

The registry is **populated at startup**; "frozen thereafter" is re-scoped by the 2026-06-20 two-tier extensibility model (G4, [10-decisions-log §2026-06-20](10-decisions-log.md)). The freeze **stays** for the primitive `Op` enum (per G3 above) and for **untrusted user-installable ops/rules** (see [09-non-goals](09-non-goals.md)). But two paths are runtime-extensible: **Tier 1 — the kernel binding table** (implementations) is *already* runtime-extensible (`extend_global_bindings`, append-only, multi-sibling; JIT-ing a kernel for an existing op identity lands here today), and **Tier 2 (new goal)** — trusted, **Fuel-orchestrated, cost-gated** registration of a **new fused-op identity** via the **declarative form** (pattern + recipe + cost carried as data; append-only registry with **stable, never-reused** `FusedOpId`s). A runtime fusion can *only* be declarative — you cannot ship Rust `fn` pointers or add enum variants at runtime — so the metadata registry's hard freeze applies to its `fn`-pointer entries, not to declarative entries Fuel registers on its own schedule.

The two layers exist because they evolve at different rates: primitives are infrastructure; fused ops are accumulating optimization wins.

## How nodes carry their op identity

A single `Op` enum carries all op identity. Most variants are primitives; one arm — `Op::Fused(id, params)` — delegates to the open registry. There is no separate `NodeKind` discriminator.

```rust
pub enum Op {
    // ~80 primitive variants, closed and exhaustive.
    Add, Sub, Mul, Div, MatMul,
    Conv2D { stride: (usize, usize), padding: (usize, usize), groups: usize },
    BroadcastTo(Shape),
    Permute(Vec<usize>),
    Slice { dim: usize, start: usize, end: usize, step: usize },
    Cast(DType),
    Const, // shared/weight data attaches via storage (see "Storage classes")
    // ... etc.

    // One arm for fused ops; parameters live in a registry-side type.
    Fused(FusedOpId, FusedOpParams),
}

pub struct Node {
    pub op:     Op,
    pub inputs: Vec<NodeId>,
    pub shape:  Shape,
    pub dtype:  DType,
}
```

- **Primitive variants** carry per-instance structural data on the variant (Conv2D's stride, Slice's geometry). Consumers handle them with specific match arms or `_` catch-alls.
- **`Op::Fused(id, params)`** is the registry-extended arm; consumers delegate uniformly to the registry or pattern-match specific fused ids.

A fused-op node and the equivalent primitive subgraph are interchangeable representations of the same computation; the optimizer moves freely between them — `decompose` going down to primitives, `pattern` coming back up. Both directions are mandatory (G1, [10-decisions-log §2026-06-20](10-decisions-log.md)); a fused op carrying only one is an opaque island the optimizer cannot move through.

## Higher-order primitives: `Op::Scan`

`Op::Scan` is Fuel's first sub-graph-carrying primitive: a bounded `lax.scan`-shaped recurrence `scan(body, init_carry, xs, consts) -> (final_carry, ys)`. It closes decision G3 ([10-decisions-log §2026-06-20](10-decisions-log.md); fulfilled [§2026-07-15](10-decisions-log.md)) — the constitution's canonical basis gap that blocked `selective_scan` and `ssd_chunk_scan` from a total `decompose`, because no pre-`Scan` primitive recipe was both total and numerically valid (an `O(seqlen)` unroll is unbounded and un-re-fusable; the `CumSum` closed-form overflows for Mamba's negative decay).

- **Encoding.** The recurrence body is carried as the node's own `inputs`, exactly like `Op::Branch`'s arms: `inputs = [init_carry, xs_0..xs_{n_xs-1}, consts.., body_new_carry, body_y]`. The two body-*exit* NodeIds are always the **last two** inputs — the convention that lets `base_map_hash`'s existing input-recursion hash the body for free, with only a params-only `op_key` arm added for `Op::Scan` itself (over `n_xs`, `bound`, `emit`, and whether `early_exit.is_some()` — never the predicate's identity). Nodes inside the body reference `Op::ScanPlaceholder { role, index }` leaves instead of real operands: `role: Carry` for the per-step carry-in (always `index: 0` — v1 is single-carry), `role: Elem, index: i` for the per-step slice of `xs[i]`. Consts are referenced by ordinary `NodeId` (they don't vary per step). `Op::ScanPlaceholder` is an inert, input-less leaf substituted only by `unroll_scan`; it is never itself realized.
- **Single carry tensor, v1.** Exactly one carry threads through the recurrence in Phase 1 — no carry tuples. Multi-carry is a possible future extension, not built.
- **Always a 2-slot bundle.** `Op::Scan` produces slot 0 = the per-step `body_y` stacked over `bound` and slot 1 = the final carry — both, always, regardless of which projection a caller asked for; `emit: ScanEmit::{All, Final}` only selects which slot the builder (`Tensor::scan`) returns by default. This follows [12-multi-output](12-multi-output.md)'s bundled-storage model.
- **`early_exit` is now a built mechanism (Phase 2).** `Op::Scan.early_exit: Option<ScanPredicate>` carries a convergence predicate; `ScanPredicate` stays a **unit marker** whose `Some`-ness signals "peel one extra trailing input." When `early_exit = Some`, the layout gains exactly one trailing slot: `inputs = [init_carry, xs.., consts.., body_new_carry, body_y, pred_exit]`. `pred_exit` is a scalar-`U8` sub-DAG over the carry (its `Op::ScanPlaceholder{Carry,0}` leaves = the pre-step carry; the shared `body_new_carry` node = the post-step carry, so `‖ξ_new − ξ_old‖ < ε` is expressible as ordinary primitives). Because it is a trailing input, `base_map_hash` / `topo_order_multi` / reachability see it for free (`op_key` folds only its presence into the hash, never the predicate's identity). The `Result`-returning builder `Tensor::scan_until` validates predicate shape/dtype/placeholders at **graph-build time**. Forward realize routes a kernel-less early-exit scan to the host step driver (`drive_scan_until_final_f32`), which stops at the runtime convergence step; `unroll_scan` peels and **ignores** the predicate (the build-time backward/oracle unroll differentiates the full static `bound` — see [04-optimization](04-optimization.md) BPTT).
- **A base-map terminal, not an executor-facing op.** No `LoweringRule` matches `Op::Scan` — it IS the primitive, its own fixpoint under `decompose` (see [The base map](#the-base-map-fully-decomposed-primitive-dag-permanently-retained) below). It carries **no native kernel** in Phase 1: the fused SSM ops still dispatch to their existing CPU/CUDA kernels on the executed path, so `Op::Scan` is reached only via explicit lowering — the verify oracle, or a kernel-absent fallback — never by production `realize`. `unroll_scan(graph, scan_id, steps) -> Result<(NodeId, NodeId), fuel_ir::Error>` (`fuel-graph/src/scan.rs`) walks the body once per step, substituting each `Op::ScanPlaceholder` with that step's carry/elem NodeId, and returns `(selected, complementary)` per `emit`. It is never registered as anyone's `decompose` — decomposing a fused op *to* `Op::Scan` is a lowering; unrolling `Op::Scan` further is a separate, opt-in materialization for verification, not part of the base map's own fixpoint.
- **Payoff is basis closure, not new runtime capability.** Closing G3 means `decompose` is now total over genuine primitives for the whole registered fused-op set (see [08-pattern-harvest](08-pattern-harvest.md)) — the base map has no more SSM-shaped opaque islands, so cost-from-decompose and re-fusion analysis cover it (see [04-optimization](04-optimization.md)). It does **not** mean SSMs run through `Op::Scan` at runtime; the fused kernels remain the executed path.
- **What Phase 2 built, and what's still deferred.** Phase 2 built the `early_exit` mechanism (above), BPTT differentiability through a scan body (unroll-then-differentiate pre-pass — see [04-optimization](04-optimization.md)), and the first non-SSM consumer (Modern Hopfield associative-memory retrieval, `hopfield_retrieve`). Still deferred: multi-carry; an `Op::Scan` native kernel (kept out on purpose — it would turn the slot-1/`last_state` OOB into a live silent read); implicit-function-theorem / deep-equilibrium gradients (BPTT is truncated to the static `bound`); a symbolic `bound`.

## Storage classes and sessions

A node is a structural fact; it does **not** need storage to exist — the model graph is built and optimized before any storage is allocated or any input arrives. Storage attaches separately, and each node has a **storage class** (how its data is shared and how long it lives), inferred from its op with an explicit override:

- **Shared** (weights/constants — `Op::Const`): one storage for the whole model, identical across all sessions; side-table keyed by `NodeId`. Loaded from the native file; written back only when it changes (e.g. training), at checkpoints.
- **Session state** (KV-caches and other per-session durable buffers — the cache-write targets, marked explicitly): one storage **per session**; side-table keyed by `(NodeId, SessionId)`. Concurrent instances of the same model are distinct `SessionId`s indexing one graph. Written back only on an explicit snapshot.
- **Transient** (activations — everything else): ephemeral scratch, allocated per realize and freed after use. Never persisted, never returned to the file. A transient value **does** cross devices mid-realize on a multi-device path (a D2D `Op::Copy`), but never crosses to disk.

This is what lets one optimized graph serve many concurrent sessions: weights shared, session state `SessionId`-indexed, activations per-realize scratch. Storage may be an **mmap-backed zero-copy view** (paged from the native file) rather than an owned buffer — the prerequisite for running models larger than RAM. See [14-lifecycle](14-lifecycle.md) for how these flow, and [11-persistence](11-persistence.md) for write-back rules.

## Layout: a side-table, not metadata on storage

Tensor layout (strides, broadcast factors, slice offsets, transpose permutations) is graph-side. A `Layout` lives in `Graph.layouts: HashMap<NodeId, Layout>`, populated automatically when a view op is added.

This decoupling makes view ops zero-copy: `Op::Transpose`, `Op::Permute`, `Op::BroadcastTo`, `Op::Reshape`, `Op::Slice`, `Op::Unsqueeze` produce nodes whose storage *aliases* their input's bytes; the view's `Layout` describes how to interpret them. No bytes allocated, no copy made.

When a kernel needs a layout it can't handle, the **optimizer** inserts an explicit `Op::Contiguize` (a materialization op that allocates new bytes in the requested layout). Kernels advertise accepted layouts via `KernelCaps` (`strided_input: bool`, etc.); the optimizer reads these and inserts fixups only when needed — never speculatively, and the executor never silently materializes.

## Data-dependent output shapes

A few ops produce an output whose size depends on runtime *values* (top-k with a fixed k is static; `NonZeroIndices`, dynamic MoE routing, and NMS produce data-dependent counts). The graph stays input-independent and node shapes stay fixed-at-construction by handling the data-dependence as a **runtime count, not a mutating shape**:

- **Default — capacity shape + valid-count.** The node's shape is a static **upper bound** (capacity); a second output carries the true count (via the multi-output bundle, [12-multi-output](12-multi-output.md)). Downstream ops honor the count as a host-scalar param — the same pattern the KV-cache uses with `cached_len`. Node shapes stay concrete; planning stays static.
- **Fallback — symbolic dim + realize-time specialization.** Where worst-case capacity is too wasteful, a bounded symbolic dim is specialized at the realize barrier among a few pre-planned size buckets. Heavier; used only when capacity waste is measured to matter.

So input-dependent *extents* are supported; what is fixed at construction is the shape *expression* (a concrete bound or a bounded symbol), not a value that mutates per input. Full design: `docs/session-prompts/data-dependent-shapes-design.md`.

## State and runtime extents in the IR

The KV-cache is the canonical instance and the seed of the general answer: a fixed-capacity buffer is allocated once, each step writes its new slice in place at a runtime offset, and that offset (the cached length) is threaded as a host-scalar parameter resolved at realize — never by mutating the node's shape. The graph structure is therefore identical across every decode step, so one optimized plan is reused token after token. This is the same discipline §Data-dependent output shapes applies to data-dependent counts: extents that vary at runtime travel as resolved parameters, the shape expression stays fixed at construction.

In-place state mutation is a first-class IR concern, expressed by destructive write ops over per-session buffers, not by hidden side effects. The recurring patterns: an append into a growing cache (the KV-cache write); a rotating/ring buffer where the write index wraps modulo a fixed capacity (a dedicated op, since modulo addressing is non-compositional); a fixed-size scratch ring (Mamba's convolution state, an ordinary in-place write); and running statistics carried back out as additional outputs (a normalization layer's exponential moving average). All share one shape: the buffer's storage class is session state, the op is destructive on its destination (the scheduler orders it after every other reader of that destination), and the mutating index travels as a runtime parameter — the structural node and its shape never change. Carrying running statistics out as outputs rather than mutating in place keeps even the stateful case inside the multi-output bundle ([12-multi-output](12-multi-output.md)).

## Tolerance metadata

Tolerance budgets attach to the IR as side-table metadata, keyed by NodeId or NodeId range. See [07-tolerance](07-tolerance.md) for the budget model; structurally:

- **Budgets are propagated, not embedded.** The optimizer reads the hierarchical specification (graph default → subgraph → per-op) when it needs a candidate rewrite's budget.
- **Per-call overrides don't mutate the IR.** A `realize(tolerance: ...)` override is communicated to the route picker, not written into the graph. The graph's tolerance is the author's intent; per-call overrides are the caller's adaptation.

## The base map: fully-decomposed primitive DAG, permanently retained

When a model is loaded or imported, the resulting DAG is whatever the source produced — a mix of primitive and fused-op nodes. This is the *user-facing form*.

Decomposition produces the **base map**: every `Op::Fused(id, params)` node is replaced by its registry `decompose`, recursively, until only primitive `Op` variants remain. The base map is the canonical, fully-elaborated form of the model.

`decompose` is **total, never-panics, and primitive→self** (per the 2026-06-20 decision G2, [10-decisions-log §2026-06-20](10-decisions-log.md); canonical home [04-optimization §`decompose` is total](04-optimization.md)). A **primitive decomposes to itself** — the recursion's fixpoint, the identity form `decompose = |_g, id, _p| id` (`fuel-graph/src/registry.rs:823`) — so the **base map is the fixpoint of `decompose` over every node** (lower until `decompose(x) == x` everywhere). A `panic!` in `decompose` is always a bug: the op is either a true primitive (must return self) or a non-primitive missing its recipe (a basis gap), distinguished by **basis membership, never by the return value**. A non-basis op that fails to decompose is a **surfaced opaque-op gap** — flagged on the base map and fed to the missing-fusion / inventory telemetry ([08-pattern-harvest](08-pattern-harvest.md)) — never a crash, never silently masquerading as a primitive. This is load-bearing for optimization itself: optimization *is* lower-to-base-map + find-best-cover, so an op that won't decompose breaks the optimizer; the recipe therefore always ships with the fused op, never deferred.

The base map is **retained as a permanent artifact**, and it is **hardware-independent and portable** — the portable *portion* of the native `.fuel` file: the thing re-optimization always restarts from, what survives transport to different hardware, and what a distribution stripper keeps. (The `.fuel` itself holds the whole graph — base map *plus* this machine's optimized paths; see [Persisting the unified graph](#persisting-the-unified-graph-base-map--optimized-paths).) Its roles:

- **Re-optimization from canonical state.** When telemetry changes (new device, memory pressure, refined profile data), re-optimizing from the base map is straightforward — it's the canonical input. Re-optimizing from the optimized graph would require undoing fusions and placements first.
- **Pattern harvesting against canonical form** (see [08-pattern-harvest](08-pattern-harvest.md)) — reading the base map avoids biasing toward optimizations that already fired.
- **Bisection during debugging** — compare an optimized run against the base map's strict execution to isolate a divergence.

The cost is roughly 2× graph storage; for typical inference graphs (low thousands of nodes) this is negligible. The architectural commitment is that the base map *is* a retained artifact.

## The optimized form: the multi-path graph (the plan IS the graph)

The base map feeds `optimize_graph` (see [04-optimization](04-optimization.md)), which transforms it **in place** into the **optimized multi-path graph**. There is no separate "plan" object — **the optimized graph is the plan.** Where the base map is a single route through the computation, the optimized graph carries **multiple alternative routes that diverge and reconverge** (different kernels, dtypes under tolerance, placements, fusions for the same logical computation), pruned to the best survivors.

Two structural facts define it:

- **Alternatives are paths, not a per-node side-table.** A region with two viable implementations (a CPU path and a CUDA path; a fused vs unfused form) appears as two sub-paths that branch from a common node and reconverge at a later common node. The points where paths diverge are **decision points** (branch points); everything between two decision points is a fixed **run** that dispatches as a unit. Decision points are **few** — most of the graph is a single agreed run — which is what lets the runtime decide rarely and dispatch long sequences rather than choosing per op (see [06-runtime](06-runtime.md)).

  These divergence/reconvergence points are not an overlay — they are a real arena node. A branch is reified as an `Op::Branch` whose merge target is an explicit reconvergence node; its alternative routes are encoded as the branch node's own inputs (each input is one route's exit), with the first route the runnability fallback the graph realizes on when no route has been picked. Because the multi-path structure is an arena fact like any other node, it compacts, graph-walks, and path-filters uniformly, and a graph with zero branch nodes is exactly a single-route graph (back-compatible by construction). Branches are built and validated at graph-build time — a malformed branch (reconvergence not downstream of the divergence, overlapping route interiors, no fallback route) is a typed build-time error, never a crash.

- **The surviving paths are a bounded Pareto frontier, per ending device.** `optimize_graph` does not keep every alternative — that explodes combinatorially (the 2026-06-14 decisions-log entry records the prototype that demonstrated this). It keeps, per ending device, the Pareto-optimal paths over the ranked axes — **one central time metric, per-tier memory, precision, accuracy** — which stays naturally small (order 10²) because those axes are kept low-dimensional (a single time metric, memory as tiers, discrete precision/accuracy). A **crowding-distance cap (keep-per-device)** is the hard backstop. The cap is **per ending device** (up to `keep` each, crowding-reduced) — never a single N across all devices, which would strand slow devices (the scalar-top-N failure mode). So ≥1 path per device survives automatically, and the total is ≤ `keep` × devices: on a 5-device machine with a fast-but-memory-heavy and a slow-but-light kernel per device, ~2 Pareto-optimal paths per device (~10 total) is the intended outcome — not "3 total." Multiple paths survive per device deliberately: a fast-but-memory-heavy path and a slow-but-light one are both kept (the runtime picks by current memory pressure), and a path may span devices (so two paths colocated at one node are *not* mergeable). Convergence merges two paths only when they are **forward-identical** from a point on; never strand the last path for a (device, backend).

Kernel binding is the last, **optional** step. Each surviving path's nodes resolve to a backend `KernelRef` via the binding-table catalog. Pre-resolving all `KernelRef`s up front takes the lookup off the hot path (best for steady-state throughput); resolving lazily at pick time (~100 ns binding-table lookup per node) minimizes time-to-first-token. Either way the binding table is a catalog, not part of the execution data structure.

(This supersedes v0.3, where the optimized form was N *separate* sibling route-graphs with every node pre-resolved, replaced wholesale by atomic Arc swap. The routes are not separate graphs — they are paths in one structure; resolution is optional, not mandatory-up-front; and re-optimization revises the multi-path graph at decision-point boundaries rather than swapping whole graphs — see [06-runtime](06-runtime.md) background re-optimization.)

## How the user-facing form, base map, and optimized graph relate

```text
user-facing form ─(decomposition)→ base map ─(optimize_graph)→ optimized multi-path graph
  (mixed prim+fused,                (primitive only,            (the PLAN: bounded Pareto
   whatever the source wrote)        canonical, PORTABLE)        paths; decision points at
                                                                 branches; per-hardware)
```

- The **user-facing form** is what Tensor ops or an import produce.
- The **base map** is the canonical primitive-only DAG — hardware-independent and portable (the portable *portion* of the `.fuel`; re-optimization's restart point; what a distribution stripper keeps). Decomposition is mechanical and deterministic: one base map per user-facing form.
- The **optimized multi-path graph** is the plan — hardware-dependent — the base map transformed in place into bounded Pareto-frontier paths with decision points at branches.

When the user calls `realize()`, the runtime waits until the plan covers the requested outputs, then dispatches the runs between decision points, choosing at each decision point among the surviving paths by current telemetry (device load, per-tier free memory) — see [06-runtime](06-runtime.md). Re-optimization (telemetry shift, new device, refined profile data) re-runs against the retained base map and adopts revised paths at decision-point boundaries; in-flight runs finish on the paths they hold.

## Persisting the unified graph (base map + optimized paths)

Because the plan IS the graph, finalizing a model after optimization persists **one** structure containing both the base map and this machine's optimized multi-path graph — there is no separate plan file in the local case.

- **Local reload is fast via scoped re-optimization.** `map_from_file` loads the whole graph. The optimized paths are tagged with the hardware/backend assumptions they were built under; on load the runtime validates them — do their kernels still resolve (by revision-hash)? have devices/backends changed? — and runs **only localized re-optimization** for what changed (a device added, a backend updated, an invalidated path), reusing every still-valid path. Same machine, unchanged backends → nearly free.
- **It works anywhere, even unstripped.** Shipping a `.fuel` carrying another machine's optimized paths is harmless: foreign hardware fails the validation, those paths drop, and re-optimization rebuilds from the always-portable base map. The only cost is a little extra file size.
- **A stripper tool removes everything but the base map** for clean distribution (smallest portable artifact). It is a convenience, not a requirement — the model runs on any machine either way.

(Full persistence model — invalidation fields, optional separate per-target caches for *shipping* pre-optimized plans, snapshots — in [11-persistence](11-persistence.md).)

## What lives where: a quick reference

| Concept | Lives in | Mutable? |
|---------|----------|----------|
| Op enum (primitives) | fuel-graph | Build-time-closed; no opaque/`Custom` node; rare additions require recompiling fuel |
| Fused-op registry (`fn`-pointer entries) | fuel-graph (metadata) + fuel-memory (BackendImpl payloads — placement open, may move to fuel-dispatch; see [02-layers](02-layers.md)), joined by `FusedOpId` | Populated at startup; `fn`-pointer entries frozen. Untrusted user ops stay frozen (G4) |
| Fused-op identities (declarative, Tier 2) | same registry, declarative entries | Trusted Fuel-orchestrated runtime registration: append-only, stable never-reused ids (G4) |
| Kernel binding table (Tier 1) | fuel-dispatch (`extend_global_bindings`) | Already runtime-extensible: append-only, multi-sibling |
| The graph (model, input-independent) | `Arc<RwLock<Graph>>`, built at load/import | Append-only; immutable after optimize |
| Base map (portable, L1) | retained primitive-only graph | Built once per model |
| Optimized multi-path graph (the plan) | the same graph, transformed in place into bounded Pareto paths + decision points | Revised by re-optimization at decision-point boundaries |
| Node storage class | inferred from op + explicit override | Fixed at build |
| Shared storage (weights) | side-table keyed by `NodeId` | One per model; write-back if changed |
| Session storage (KV etc.) | side-table keyed by `(NodeId, SessionId)` | Per session; write-back on snapshot |
| Transient storage (activations) | per-realize scratch | Ephemeral; never persisted |
| Layout side-table | `Graph.layouts` | Auto-populated on view-op push |
| Tolerance metadata | side-table keyed by NodeId / range | Set at build, immutable thereafter |
| Per-path KernelRef | side-table per path | Optionally pre-resolved; else lazy at pick time |

Implementation detail (specific Rust types, file layout, commit mechanics) lives in phase docs. This section pins the architectural shape: the graph is the input-independent model; the plan is that graph optimized in place into bounded Pareto paths with few decision points; storage attaches by class and session; side-tables key everything that varies across paths or sessions.
