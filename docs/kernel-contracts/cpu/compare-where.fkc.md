# fuel-cpu-backend — compare / where kernel contracts

Portable CPU elementwise comparison and ternary-select kernels (the `compare-where` family) from
`fuel-cpu-backend/src/chassis/compare.rs` and `fuel-cpu-backend/src/byte_kernels.rs`. Two shapes
live here:

- **Comparison** (`Eq/Ne/Lt/Le/Gt/Ge` × `{f32,f64,bf16,f16}`) — typed `T × T` inputs, **U8 output
  mask** (1 where the predicate holds else 0). One chassis function `compare<T,U>`
  (`chassis/compare.rs:94`) backs all 24 per-(op,dtype) thunks generated by `compare_thunk!`
  (`byte_kernels.rs:233`). Output is *narrower* than the inputs, so the chassis validates **element
  counts**, not byte counts (`compare.rs:107`).
- **Where / ternary select** (`where_kernel!`, `byte_kernels.rs:288`) — `(cond, a, b) → out` with
  `cond` U8 and `a/b/out` sharing dtype `T`.

Both are **contiguous-only, zero-offset, row-major** with **no broadcasting**: the pipelined
executor's auto-Contiguize pass realizes any strided/broadcast/offset operand into a dense buffer
*before* these kernels run (inventory "Cross-cutting facts"). Half floats (`bf16`/`f16`) compare by
widening each operand to **f32** (lossless, order-preserving) and applying the predicate; this is
bit-identical to a native half comparison for finite values and identical for NaN. NaN follows
IEEE-754: every ordered comparison with a NaN operand is `false` → `0`, **except** `Ne(NaN, NaN)`
which is `true` → `1` (`compare.rs:129-171`). All validation returns `Result` (element-count
mismatch → `Error`), never panics on the production path.

This bundle authors **26 sections**: the `compare` chassis family kernel, the 24 per-(op,dtype)
compare thunks, and `where_kernel`. Every cost block is `provenance: declared` (an author prior the
Judge refines, §4.4) — no fabricated measured numbers. All are bandwidth-bound elementwise kernels:
the FLOPs term is one comparison/select per output element (`n`), and `bytes_moved` is the literal
read+write traffic, which differs between compare (U8 output) and where (4 buffers).

## compare  (Eq / Ne / Lt / Le / Gt / Ge — chassis family kernel)

One-line: the elementwise-compare chassis function shared by all 24 per-(op,dtype) thunks; `T × T`
in, U8 mask out.

`compare<T,U>` (`chassis/compare.rs:94`) is the single logical kernel that every comparison thunk
delegates to. It takes three `CpuStorageBytes` views — `lhs`, `rhs` (typed `T`) and `output` (typed
`u8`) — validates that the three have **equal element counts** (not byte counts; the U8 output is
narrower than `T ≠ u8` inputs, `compare.rs:107`), then walks positionally
`out[i] = U::apply(lhs[i], rhs[i])`, where `U::apply` returns `1u8` when the predicate holds else
`0u8` (`compare.rs:116-118`). The op (`Eq/Ne/Lt/Le/Gt/Ge`) is selected by the type parameter `U:
CompareOp<T>`; the per-precision predicate is implemented once on `CompareOpCore` (an `f32` and an
`f64` method) and the four dtype impls fall out of blanket impls (`compare.rs:61-83`). bf16/f16
route through `to_f32()` (`compare.rs:73-82`). No broadcasting, no strided/offset awareness: the
buffer base **is** element 0. Numerics are exact IEEE-754 ordered comparisons (the bool→u8 map is
the only post-processing), so the kernel is bitwise-deterministic on any hardware. This is a
"logical family" section: it carries the contract shared by all 24 thunks below; each thunk pins one
`(op, dtype)` cell of the dispatch key. Algorithm/perf: a single bandwidth-bound positional loop, no
accumulation, no allocation (output pre-allocated by the executor). Limitation: contiguous-only —
any non-contiguous operand must be contiguized by the planner first.

## where_kernel  (ternary select: U8 cond + T x T -> T)

One-line: branchless ternary select `out[i] = if cond[i] != 0 { a[i] } else { b[i] }`; `cond` U8,
`a`/`b`/`out` share dtype `T`.

`where_kernel!` (`byte_kernels.rs:288`) generates four per-dtype thunks `where_{f32,f64,bf16,f16}`
(`byte_kernels.rs:320-323`). Each takes **four** `CpuStorageBytes` views — `cond` (U8), `a`, `b`
(typed `T`) and `out` (typed `T`) — validates that all four have **equal element counts**
(`byte_kernels.rs:301-311`; the U8 cond is narrower than `T` so element counts, not byte counts, are
checked) then walks `out[i] = if cond[i] != 0 { a[i] } else { b[i] }` (`byte_kernels.rs:312-314`).
This is the natural consumer of the U8 masks the comparison family above produces. **Any non-zero
cond byte selects `a`** (it is `!= 0`, not `== 1`). No broadcasting, no strided/offset awareness
(contiguous, zero-offset). There is **no arithmetic and no dtype conversion** — `a[i]`/`b[i]` are
copied verbatim, so the operation is exact for every dtype including bf16/f16 (no f32 round-trip;
unlike the compare family, where halves are widened, select just copies the chosen lane). Output is
dtype `T`. Per the as-built dispatch convention the binding dtype list is `[U8, T, T, T]` (cond, a,
b, out). Algorithm/perf: a single bandwidth-bound positional loop with one branch per element (one
read of cond + one read of the selected operand + one write); no allocation (output pre-allocated).
Limitation: contiguous-only — strided/broadcast/offset operands must be contiguized first.
