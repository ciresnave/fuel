# fuel-cpu-backend — elementwise-binary kernel contracts

The portable CPU elementwise-binary family: `out[i] = op(lhs[i], rhs[i])` over two
contiguous, zero-offset, row-major buffers of equal element count. Eight ops
(Add / Sub / Mul / Div / Maximum / Minimum / Pow / Rem) × four dtypes
(F32 / F64 / BF16 / F16). One shared chassis `binary<T, U>`
(`fuel-cpu-backend/src/chassis/binary.rs:78`); per-`(op, dtype)` public thunks generated by
`binary_thunk!` (`fuel-cpu-backend/src/byte_kernels.rs:54`).

**Family-wide facts (every kernel below shares these unless its section says otherwise):**

- **Layout: contiguous-only, zero-offset, row-major, no broadcasting.** The chassis validates
  byte lengths `lhs.len_bytes() == rhs.len_bytes() == out.len_bytes()`
  (`chassis/binary.rs:88-96`) then walks a pure positional loop `out[i] = U::apply(lhs[i], rhs[i])`
  (`chassis/binary.rs:100-102`). It never consults a `Layout`/strides/offset. The five-flag
  layout capability is therefore `{contiguous: required, strided: rejected, broadcast_stride0:
  rejected, start_offset: rejected, reverse_strides: rejected}` on **both** operands — the
  pipelined executor's auto-Contiguize pass realizes any strided / broadcast / offset input into a
  dense buffer *before* these kernels run (`awkward_layout_strategy: requires_contiguous`).
  Shapes must already be identical and contiguous (no NumPy/PyTorch broadcasting in the kernel).
- **Dtype monomorphization.** Each thunk is one `(op, dtype)` pair. `f32` / `f64` evaluate
  natively (`BinaryOpCore::{f32,f64}`); `bf16` / `f16` widen each element to **f32**, do the math,
  and narrow back on store (the blanket `BinaryOp<half::{bf16,f16}>` impls,
  `chassis/binary.rs:57-67`). There is no half-native arithmetic path.
- **Output: pre-allocated, fully overwritten.** The output buffer is caller-allocated to the
  exact byte size; the kernel overwrites every element (no read of prior contents, no aliasing
  with the inputs). Output dtype = input dtype, output shape = input shape, output layout =
  contiguous row-major.
- **Validation is build/runtime byte-length checks returning `Result`** (never panics on the
  production path); the error message carries the thunk name and the three observed byte lengths.

**Cost provenance.** Every kernel in this family marks `cost.provenance: judge_measured` — the
Judge bootstraps and refines the measured cost for this CPU surface. Where genuinely derivable
from the op the cost block carries the **formula hints** an elementwise kernel admits — it is
bandwidth-bound: it touches `n` output elements, reading `lhs` + `rhs` and writing `out`
(`bytes_moved ≈ 3 · n · dtype_bytes`), with one scalar op per element (`flops ≈ n`). These hints
are structural facts of the loop, not fabricated coefficients; the launch overhead and any
absolute timing are left for the Judge to populate (no placeholder numbers).

**Precision.** `f32` / `f64` are the native IEEE-754 operation, bit-stable on the same hardware;
`bf16` / `f16` are bit-stable on the same hardware via the deterministic f32 round-trip
(widen → op → narrow). Per-op IEEE semantics (Div inf/NaN; Max/Min NaN-as-missing; Pow `powf`;
Rem floored) are noted per section. All bounds are author-declared seeds the Judge audits.

## binary  (Add / Sub / Mul / Div / Maximum / Minimum / Pow / Rem — the shared chassis)

Elementwise binary arithmetic and extremum over a contiguous, same-shape operand pair; half
floats via an f32 round-trip.

This is the umbrella entry for the one logical chassis kernel `binary<T, U>`
(`chassis/binary.rs:78`) that backs all 32 per-`(op, dtype)` thunks below. It does a positional
walk `out[i] = op(lhs[i], rhs[i])` over contiguous, zero-offset, row-major buffers — **no
broadcasting** (the chassis validates `lhs.len_bytes() == rhs.len_bytes() == out.len_bytes()` and
walks element-for-element). The op is selected by the `BinaryOp<T>` marker (`Add`, `Sub`, `Mul`,
`Div`, `Maximum`, `Minimum`, `Pow`, `Rem`), the dtype by the monomorphized `T`. `f32` / `f64`
evaluate natively; `bf16` / `f16` widen to f32, compute, and narrow on store. Each concrete op
maps to a distinct Fuel `OpKind` (`AddElementwise`, `SubElementwise`, `MulElementwise`,
`DivElementwise`, `MaximumElementwise`, `MinimumElementwise`, `PowElementwise`, `RemElementwise`),
so this umbrella section is documentation of the shared algorithm; the registrable contracts are
the per-`(op, dtype)` sections that follow (each names its own `op_kind` + `entry_point`). Known
limitation: contiguous-only — any strided / broadcast / non-zero-offset operand must be
contiguized by the planner first (`requires_contiguous`).

```fkc
kernel: binary

blurb: "Elementwise binary arithmetic/extremum; contiguous same-shape; half via f32."
backend: Cpu
kernel_source: "portable-cpu"
entry_point: "fuel_cpu_backend::chassis::binary::binary"   # the shared chassis fn; §12.6
kernel_revision_hash: auto

accept:
  inputs:
    - name: lhs
      dtypes: [F32, F64, BF16, F16]
      layout: { contiguous: required, strided: rejected, broadcast_stride0: rejected, start_offset: rejected, reverse_strides: rejected }
      rank: any
      shape_constraint: same_as=rhs
    - name: rhs
      dtypes: [F32, F64, BF16, F16]
      layout: { contiguous: required, strided: rejected, broadcast_stride0: rejected, start_offset: rejected, reverse_strides: rejected }
      rank: any
      shape_constraint: same_as=lhs
  op_params: { variant: None }

return:
  outputs:
    - name: out
      dtype_rule: passthrough(lhs)
      shape_rule: same_as(lhs)
      layout_guarantee: contiguous
      aliasing: none

caps:
  awkward_layout_strategy: requires_contiguous    # planner inserts Op::Contiguize (itself an FKC kernel) + sums its cost
  fast_paths:
    - { when: "all_inputs_contiguous", class: cheap_elementwise }
  in_place: false
  alignment_bytes: 64
  access_granularity_bits: 8

cost:

  class: cheap_elementwise
  flops: "n"                            # one scalar op per element (n = output element count)
  bytes_moved: "3 * n * dtype_bytes"    # read lhs + rhs, write out — bandwidth-bound
  memory: { device_bytes: 0, host_bytes: "n * dtype_bytes", disk_bytes: 0 }

precision:
  bit_stable_on_same_hardware: true
  max_ulp: ~
  max_relative: ~
  max_absolute: ~
  audited: true
  notes: "f32/f64 native IEEE op; bf16/f16 widen to f32 then narrow on store. Per-op IEEE semantics vary (see concrete sections)."

determinism: same_hardware_bitwise
```
