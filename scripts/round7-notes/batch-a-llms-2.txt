Five binaries migrated, five skipped on documented infrastructure gaps. Pattern: any lazy model whose `LayerWeights` shape matches LLaMA's (token_embedding: Arc<[f32]>, Vec<LayerWeights>, final_norm_gain, output: WeightStorage) can reuse `LlamaWeights::load_from_mmapped` by lifting fields directly into the model-specific weights struct — used by mistral, helium, and granite migrations. Built a small per-binary HF-config deserializer for each, since none of these lazy modules expose their own from_hf_json_str. llama2-c migrated to inference-only via `load_llama2c_bin_path`; train/eval subcommands bail with a clear message because they depend on eager fuel-nn loss + fuel-datasets, neither of which has a lazy migration. Removed the now-unneeded `mod training;` (deleted training.rs) and dropped the `required-features = ["fuel-datasets"]` constraint from Cargo.toml (the lazy port no longer needs the dataset crate, and that feature's mnist code is currently broken). Fixed an existing build error in llama/main.rs (`token_generated` type inference for `saturating_sub`). Skipped binaries (gemma4, glm4, granitemoehybrid, jina-bert, modernbert) all need bespoke per-model safetensors loaders in their lazy modules before their binaries can be migrated — each has a different LayerWeights shape that doesn't reuse the LLaMA layout. These should be tackled by either landing model-specific Weights::load_from_mmapped in the lazy modules, or by promoting the existing `pub(crate) load_tensor_as_f32`/`load_transposed_matrix_preserve_dtype` helpers from `fuel-core::lazy` to `pub` so binaries can assemble custom loaders themselves.
