skipped: custom-ops (eager-only infrastructure: lazy CustomOp1 trait not shipped. The example exists to demonstrate how to write a user-defined CPU+CUDA kernel via the eager CustomOp1 trait + storage.as_any().downcast_ref::<CpuBackendStorage>() / CudaStorage. No lazy equivalent (no LazyTensor::apply_op1 / no LazyCustomOp1) exists in fuel-core::lazy.)
  - mnist-training (eager-only infrastructure: lazy Conv2d layer in fuel-core::lazy_nn. fuel-core::lazy_nn currently exports only LazyLinear + LazyEmbedding; no LazyConv2d / LazyDropout / LazyMaxPool2d, so the ConvNet model path cannot be migrated. The Linear-only path is also blocked because fuel_datasets::vision::Dataset returns eager Tensors (m.train_images: Tensor) with no lazy bridge — the host loop would have to realize-and-rebuild every batch via Tensor::to_vec1::<f32>() ↔ LazyTensor::from_f32, which is a separate dataset port (not yet shipped).)
  - reinforcement-learning (eager-only infrastructure: gym_env / vec_gym_env / dqn / ddpg / policy_gradient sub-modules all use eager Tensor + fuel_nn::{seq, Activation, AdamW, VarMap, VarBuilder, linear, Optimizer} + Module::forward(&Tensor). The lazy SGD/AdamW (fuel-core::lazy_nn_optim::LazyOptimizer + LazyVar) exists but there is no lazy sequential / Activation / Module surface to drive a policy net, no lazy bridge for the per-step gym observations (returned as eager Tensors from gym_env::step), and no lazy categorical-sample / distribution helper for action selection.)
  - llama_multiprocess (eager-only infrastructure: lazy multi-GPU + NCCL sharded VarBuilder. The example needs fuel_nn::var_builder::ShardedSafeTensors + baracuda_nccl::Communicator to shard a 70B LLaMA across multiple GPUs. No lazy ShardedSafeTensors / lazy NCCL-aware LayerWeights exists; Llama2cModel::from_hub is single-device only. Migration would also need a lazy version of model.rs (custom NCCL-aware Llama).)
  - orpheus (eager-only infrastructure: lazy SNAC integration with a Llama-family TTS pipeline. The example chains LlamaConfig::into_config(use_flash_attn) → Llama + Cache → token loop → fuel_transformers::models::snac::Model::decode(&[&codes0, &codes1, &codes2]) which returns the audio PCM. lazy_snac is shipped (fuel-core::lazy_snac) but does not yet expose a multi-codebook .decode([&LazyTensor; 3]) entry point that matches Orpheus' 7-tokens-per-frame interleaved codes0/1/2 layout; orpheus also uses a custom audio-token-to-codebook permutation (audio_tokens.chunks_exact(7)) glued to the Llama generation loop that has no lazy parallel. Migrating cleanly needs a multi-model lazy composition (LazyTensor LLaMA logits → custom audio-codebook permute → LazySnac::decode_3_codebooks) that hasn't been built.)

Batch f migrated 5 / 10 binaries.

Patterns that emerged:
1. Llama2cModel::from_hub + LlamaTokenizer::from_hub + generate_streaming_with_kv_context is the canonical LLM migration template (used for qwen, would also fit mistral / phi / llama / gemma host-of-LLMs). Auto-detects EOS via the standard <|im_end|> / </s> / <|eot_id|> / <|end_of_text|> probe. SamplingStrategy::Temperature carries the temp+seed; eos_id falls out of LlamaTokenizer.
2. LazyOnnxEval is a drop-in replacement for fuel_onnx::simple_eval, with three big practical changes: (a) inputs/outputs are LazyTensor (HashMap<String, LazyTensor>); (b) all sibling inputs must live on the same graph as the first input — the trick is to create a tiny F32 anchor LazyTensor and then build every other input via anchor.const_*_like(...); (c) state that persists across iterations is round-tripped through Vec<f32> (.realize_f32()) and rebuilt as a fresh const each iteration.
3. ONNX example post-processing that used eager Tensor scalar broadcast (let s = Tensor::try_from(255.0f32)? ; out * s) is rewritten as host-side scaling on the realized vec; for the EsrGan branch I still bridge back through fuel::Tensor::from_vec at the very end because save_image takes &Tensor.

Pre-existing fuel-transformers compile failure: deepseek2.rs nonzero impl was missing the HostBuffer::I8 arm. Patched (one-line) to unblock the cargo build of the example crate; this is an eager-source bug that predates the batch (commit 082d7ffa landed I8 into HostBuffer without updating this match).
